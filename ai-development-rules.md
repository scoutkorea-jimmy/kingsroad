# AI Development Rules

## 목적

이 문서는 `뱅기노자(BANGINOJA)` 프로젝트를 함께 개발하는 모든 AI를 위한 공용 작업 규칙이다.  
ChatGPT, Claude, 그리고 앞으로 추가될 수 있는 다른 AI 도구들도 모두 이 문서를 먼저 읽고 작업해야 한다.

## 최상위 원칙

1. 모든 AI는 작업 시작 전에 반드시 이 문서를 읽는다.
2. 모든 AI는 `project-priority-table.md`를 확인하고 현재 작업의 우선순위를 이해한 뒤 개발한다.
3. 모든 수정사항은 가능하면 커밋, 푸시, 배포 반영까지 진행하여 사용자가 실제 결과를 검토할 수 있게 한다.
4. 설명이 필요한 경우, 개발자인 박지민이 비개발자라는 점을 기준으로 최대한 쉽게 설명한다.
5. 중요한 작업은 시작 전에 계획 문서를 만들고, 작업 완료 후 더 이상 필요 없으면 해당 계획 문서를 삭제한다.
6. KMS와 AI 작업 가이드는 항상 같이 갱신한다.

## 프로젝트 우선순위 원칙

개발은 반드시 `project-priority-table.md`의 우선순위를 따른다.

1. 인증/권한
2. 데이터 저장 구조
3. 관리자 발행물과 공개 페이지 연결
4. 커뮤니티 실서비스화
5. 관리자 운영 기능 고도화
6. 투어 예약
7. 책 주문/결제
8. 부가 기능

사용자가 명시적으로 승인하지 않는 한, AI는 이 순서를 임의로 건너뛰지 않는다.

## 작업 시작 전 필수 절차

모든 AI는 작업 시작 전에 아래 순서를 따른다.

1. `ai-development-rules.md`를 읽는다.
2. `project-priority-table.md`를 읽는다.
3. `kms.md`를 읽어 현재 규칙, 현황, 버전 원칙을 확인한다.
4. 필요한 경우 작업 계획 문서를 만든다.
5. 그 다음에야 코드 또는 문서를 수정한다.

## 작업 계획 문서 규칙

1. 기능 구현, 구조 변경, 운영 규칙 변경처럼 범위가 있는 작업은 시작 전에 계획 문서를 만든다.
2. 계획 문서 파일명은 영어로 작성한다.
3. 계획 문서에는 최소한 아래 내용을 포함한다.
   - 작업 목적
   - 수정 대상 파일
   - 예상 영향 범위
   - 완료 기준
4. 작업이 완료되면 해당 계획 문서는 삭제한다.
5. 장기적으로 남겨야 하는 내용은 계획 문서가 아니라 `kms.md` 또는 다른 기준 문서로 옮긴다.

## 비개발자 설명 규칙

개발자인 박지민은 비개발자이므로, 설명이 필요할 때는 아래 기준을 따른다.

1. 어려운 용어를 그대로 쓰지 말고 쉬운 말로 먼저 설명한다.
2. “무엇이 바뀌는지”, “왜 필요한지”, “어떤 영향이 있는지”를 먼저 말한다.
3. 기술적 세부사항은 필요할 때만 덧붙인다.
4. 문제가 생기면 원인보다 먼저 체감되는 현상을 설명한다.
5. 완료되지 않은 기능은 현재 어디까지 됐는지 분명하게 설명한다.

## 파일명 및 네이밍 규칙

1. 파일명은 영어를 기본 원칙으로 한다.
2. 새로 만드는 기획 문서, 정책 문서, 운영 문서는 영어 파일명을 사용한다.
3. 파일명은 짧고, 예측 가능하고, 의미가 분명해야 한다.
4. 기존 파일명 변경은 영향 범위를 설명할 수 있을 때만 진행한다.

## 기준 문서 규칙

1. 구현 상태의 1차 기준은 저장소 안의 실제 코드다.
2. 개발 순서의 기준 문서는 `project-priority-table.md`다.
3. 운영 기준의 중심 문서는 `kms.md`다.
4. AI 협업 기준의 중심 문서는 `ai-development-rules.md`다.
5. 관리자 KMS와 관리자 버전 기록은 실제 프로젝트 상태를 반영해야 한다.
6. 문서와 코드가 충돌하면, 가능하면 같은 작업 안에서 둘 중 하나를 반드시 갱신한다.

## AI 간 협업 규칙

1. 구조 변경을 제안하기 전에 반드시 기존 코드를 먼저 읽는다.
2. 다른 AI가 만든 작업을 사용자가 요청하지 않은 한 덮어쓰거나 되돌리지 않는다.
3. 작업 트리가 이미 더러우면, 자신의 변경만 분리해서 다루고 관련 없는 변경은 건드리지 않는다.
4. 작업을 넘길 때는 아래 내용을 남긴다.
   - 무엇을 바꿨는지
   - 무엇이 아직 남았는지
   - 어떤 가정을 했는지
   - 어떤 리스크가 있는지
5. 다른 AI가 이미 만든 규칙이나 구조가 있으면 중복 생성보다 확장 우선으로 간다.
6. 가능하면 같은 파일을 여러 AI가 동시에 수정하지 않는다.

## 구현 규칙

1. 화면만 있고 실제 저장/처리가 없으면 “구현 완료”라고 표현하지 않는다.
2. 새 사용자 기능을 만들 때는 최소한 아래를 생각한다.
   - 어디서 진입하는지
   - 상태를 어디서 들고 있는지
   - 새로고침 후에도 유지되는지
   - 실패 시 어떻게 보이는지
3. 관리자 기능은 필요한 경우 공개 페이지와 연결되는지 반드시 확인한다.
4. 인증 관련 변경은 최소한 아래를 함께 본다.
   - 로그인 상태 유지
   - 로그아웃 처리
   - 권한 체크
   - 비인가 접근 처리
5. 데이터 기능은 아래 셋 중 무엇인지 분명히 구분한다.
   - 임시 `localStorage`
   - 목업 데이터
   - 실제 운영 가능한 영속 데이터

## 버전 관리 규칙

버전은 반드시 `AA.BBB.CCC` 형식으로 관리한다.

1. `AA`
   - 대버전
   - 사용자가 직접 판단하고 올린다
2. `BBB`
   - 기능 추가 또는 기능 삭제처럼 눈에 보이는 기능 단위 변경 시 올린다
3. `CCC`
   - 마이너 수정, 문구 수정, 버그 수정, 작은 개선 시 올린다

AI는 버전을 임의로 크게 올리지 않는다.  
기능 추가/삭제가 아니면 `BBB`를 올리지 않고, 작은 수정은 `CCC`만 조정한다.

## 관리자 페이지 버전 기록 규칙

관리자 페이지에는 버전 기록이 반드시 관리되어야 하며, 각 버전은 아래 구조를 따른다.

1. 버전 번호
2. 날짜
3. 200자 내외의 핵심 수정사항 요약
4. 그 아래 항목별 세부 업데이트 내역

세부 업데이트 내역은 사용자가 한눈에 파악할 수 있도록 기능별로 분리해서 적는다.

예시 구조:

- 버전: `01.002.003`
- 핵심 수정사항: 메인 홈 강연 일정 노출, 로그아웃 기능 추가, 마이페이지 기본 화면 신설
- 세부 내역:
  - 홈: 왕사남 강연 일정 섹션 추가
  - 인증: 로그아웃 처리 및 로그인 상태 유지 반영
  - 계정: 마이페이지 신설

## KMS 동기화 규칙

1. 새로운 개발 규칙이 생기면 `ai-development-rules.md`와 `kms.md`를 함께 갱신한다.
2. 중요한 기능 구현이 끝나면 `kms.md`의 구현 현황도 함께 갱신한다.
3. 디자인 테마가 바뀌면 `kms.md`의 디자인 항목을 함께 수정한다.
4. 버전이 바뀌면 관리자 페이지 버전 기록과 `kms.md`를 함께 맞춘다.
5. KMS 내용을 수정할 때는 단순 결과만 적지 말고, 수정하게 된 계기와 배경도 함께 남긴다.
6. KMS의 수정 기록은 나중에 다른 AI나 사람이 맥락을 보고 이어서 개발할 수 있도록 작성한다.

## UI 및 디자인 규칙

1. 현재 `뱅기노자`의 시각 방향을 유지한다.
   - 조선 왕실의 분위기
   - 편집 디자인 같은 진중함
   - 절제된 금색 포인트
   - 다크 기반의 프리미엄 톤
2. 브랜드 결을 흐리는 평범한 스타트업형 UI를 함부로 넣지 않는다.
3. 새 섹션은 기존의 타이포, 간격, 상징 언어와 어울려야 한다.
4. 중요한 시각 변화는 KMS나 버전 기록에도 반영한다.

## Git 및 배포 규칙

1. 의미 있는 코드 변경 후에는 반드시 `git status`를 확인한다.
2. 관련 있는 파일만 커밋한다.
3. `.DS_Store` 같은 환경 잡파일은 커밋하지 않는다.
4. 사용자에게 보이는 변경이 완료되면 가능하면 푸시까지 진행한다.
5. GitHub Pages나 프리뷰 배포를 쓰는 경우, 푸시 후 배포 반영 여부까지 확인한다.
6. 푸시나 배포가 실패하면, 막힌 원인을 정확히 기록한다.
7. 배포가 되지 않은 수정사항은 “검토 가능” 상태로 보지 않는다.

## 보안 및 안전 규칙

1. 시크릿, 토큰, 실제 비밀번호를 저장소 파일에 넣지 않는다.
2. 운영용 자격증명을 임의로 만들어내지 않는다.
3. 현재 인증이 데모 수준이면, 실제 인증처럼 보이게 설명하지 않는다.
4. 개인정보, 결제, 관리자 기능은 보수적으로 변경하고 주석이나 문서 설명을 남긴다.

## 작업 종료 전 체크리스트

중요 작업을 마치기 전에는 아래를 확인한다.

- 현재 우선순위와 맞는 작업이었는가
- `ai-development-rules.md`를 먼저 확인하고 시작했는가
- `kms.md`를 같이 갱신해야 하는 작업이었는가
- 낡은 문서를 같이 갱신했는가
- 목업 동작을 분명히 표시했는가
- `git status`를 확인했는가
- 커밋, 푸시, 배포까지 고려했는가
- 완료된 계획 문서를 삭제했는가
- 끝내지 못한 이유가 있다면 명확히 기록했는가
