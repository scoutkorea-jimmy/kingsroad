# KMS

## 목적

이 문서는 `왕사들(WANGSADEUL)` 프로젝트의 운영 기준 문서다.  
현재 구현 현황, 개발 규칙, 버전 관리 원칙, 디자인 방향, AI 협업 기준을 한 곳에서 확인할 수 있어야 한다.

## 문서 운영 원칙

1. `kms.md`와 `ai-development-rules.md`는 같이 관리한다.
2. 새 규칙이 추가되면 두 문서를 동시에 갱신한다.
3. 구현 현황이 바뀌면 KMS도 함께 갱신한다.
4. 관리자 페이지의 KMS 화면은 이 문서와 같은 기준을 반영해야 한다.
5. KMS를 수정할 때는 무엇이 바뀌었는지만 적지 말고, 왜 수정했는지 계기와 배경도 함께 기록한다.
6. KMS 기록은 나중에 다른 AI나 사람이 보고도 맥락을 이해할 수 있어야 한다.

## 현재 우선순위 기준

기준 문서: `project-priority-table.md`

1. 인증/권한
2. 데이터 저장 구조
3. 관리자 발행물과 공개 페이지 연결
4. 커뮤니티 실서비스화
5. 관리자 운영 기능 고도화
6. 투어 예약
7. 책 주문/결제
8. 부가 기능

## AI 작업 기본 원칙

1. 모든 AI는 작업 시작 전에 `ai-development-rules.md`를 먼저 읽는다.
2. 모든 AI는 `project-priority-table.md`를 확인하고 우선순위 안에서 작업한다.
3. 모든 수정사항은 가능하면 푸시와 배포까지 진행하여 실제 검토 가능 상태로 만든다.
4. 설명이 필요하면 비개발자도 이해할 수 있는 표현을 사용한다.
5. 계획이 필요한 작업은 문서로 만들고, 완료 후 삭제한다.

## 버전 관리 원칙

버전 형식은 `AA.BBB.CCC`를 사용한다.

1. `AA`
   - 대버전
   - 사용자가 직접 판단해서 올린다
2. `BBB`
   - 기능 추가 또는 삭제 시 올린다
3. `CCC`
   - 작은 수정, 버그 수정, 문구 수정, 마이너 개선 시 올린다

## 관리자 페이지 버전 기록 기준

관리자 페이지의 버전 기록은 아래 구조를 따른다.

1. 버전 번호
2. 날짜
3. 200자 내외 핵심 수정사항
4. 세부 업데이트 내역

세부 업데이트 내역은 기능별로 나누어 한눈에 볼 수 있어야 한다.

예시:

- 버전: `01.002.003`
- 핵심 수정사항: 메인 홈 강연 일정 추가, 로그아웃 기능 구현, 마이페이지 기본 구조 추가
- 세부 업데이트:
  - 홈: 왕사남 강연 일정 노출
  - 인증: 로그아웃 및 로그인 상태 유지
  - 계정: 마이페이지 생성

## KMS 기록 방식

KMS를 수정할 때는 가능하면 아래 구조를 따른다.

1. 무엇이 바뀌었는가
2. 왜 바꾸게 되었는가
3. 어떤 배경과 맥락이 있었는가
4. 이후 어떤 작업과 연결되는가

짧은 주석이나 변경 기록 형태여도 괜찮지만, 나중에 맥락 없이 봐도 이해 가능해야 한다.

## 구현 현황 메모

현재 기준으로 아래 항목은 운영 문서와 연결해서 관리해야 한다.

1. 관리자 KMS 페이지 기본 UI 반영 완료
   - 계기: 저장소 문서와 관리자 화면의 기준이 서로 어긋나면 운영 중 혼선이 생길 수 있었음
   - 배경: 여러 AI가 함께 개발하는 구조에서, 관리자 화면에서도 같은 규칙을 바로 확인할 수 있어야 했음
2. 관리자 버전 기록 UI 기본 반영 완료
   - 계기: 수정사항을 한눈에 파악할 수 있는 운영 화면이 필요했음
   - 배경: 핵심 요약과 세부 업데이트 내역, 그리고 수정 배경까지 함께 보는 구조가 요구되었음
3. 버전 체계를 `AA.BBB.CCC` 형식으로 코드에 반영 완료
   - 계기: 버전 기준이 프로젝트 전반에서 일관되지 않았음
   - 배경: 기능 추가/삭제와 마이너 수정의 차이를 명확히 나눠 관리할 필요가 있었음
4. 배포 완료 상태 확인 절차는 계속 필요
   - 계기: 수정 후 실제 검토 가능 상태를 보장하려면 배포 확인까지 필요함
   - 배경: 현재 GitHub 인증 문제로 자동 푸시가 막힐 수 있으므로, 배포 확인 절차를 계속 운영 규칙으로 유지해야 함
