# KMS

## 목적

이 문서는 `왕사들(WANGSADEUL)` 프로젝트의 운영 기준 문서다.  
현재 구현 현황, 개발 규칙, 버전 관리 원칙, 디자인 방향, AI 협업 기준을 한 곳에서 확인할 수 있어야 한다.

## KMS 제1 원칙

KMS의 제1 기능은 `기능정의서`다.  
즉, 이 문서는 다른 개발자가 코드를 직접 열지 않아도 현재 홈페이지가 어떤 기능을 갖고 있고, 각 기능이 어디까지 구현되어 있으며, 무엇이 아직 남아 있는지 먼저 이해할 수 있어야 한다.

현재 기능정의서는 `실제 라우트`, `실제 노출 콘텐츠`, `실제 저장소 구조`를 기준으로 작성한다.  
기획상 있었지만 화면에서 제거된 영역은 현재 기능으로 기록하지 않는다.

## 문서 운영 원칙

1. `kms.md`와 `ai-development-rules.md`는 같이 관리한다.
2. 새 규칙이 추가되면 두 문서를 동시에 갱신한다.
3. 구현 현황이 바뀌면 KMS도 함께 갱신한다.
4. 관리자 페이지의 KMS 화면은 이 문서와 같은 기준을 반영해야 한다.
5. KMS를 수정할 때는 무엇이 바뀌었는지만 적지 말고, 왜 수정했는지 계기와 배경도 함께 기록한다.
6. KMS 기록은 나중에 다른 AI나 사람이 보고도 맥락을 이해할 수 있어야 한다.

## 현재 우선순위 기준

기준 문서: `project-priority-table.md`

1. 인증/권한
2. 데이터 저장 구조
3. 관리자 발행물과 공개 페이지 연결
4. 커뮤니티 실서비스화
5. 관리자 운영 기능 고도화
6. 투어 예약
7. 책 주문/결제
8. 부가 기능

## P1 진행 상태

현재 기준으로 P1은 `현재 정적 배포 구조 안에서 완료` 상태다.

### 현재까지 완료된 P1

1. 인증/권한
   - 완료
   - `WSD_AUTH` 기준으로 로그인, 회원가입, 로그아웃, 세션 유지가 한 구조에서 동작함
2. 관리자 발행물과 공개 페이지 연결
   - 완료
   - 관리자 발행 칼럼이 공개 칼럼 페이지와 홈 화면에 반영되도록 연결됨
3. 데이터 저장 구조
   - 완료
   - `WSD_DB`, `WSD_STORES`, `WSD_SAVE` 기준으로 회원, 세션, 게시글, 댓글, 칼럼, 권한 엔티티 구조를 분리함

### P1 완료 처리 이유

1. 현재 운영 환경은 GitHub Pages 기반 정적 배포다.
2. 따라서 P1의 핵심은 외부 서버 유무보다도, 인증과 저장 구조가 임시 목업이 아니라 명확한 계층으로 분리되어 있는가에 있다.
3. 이번 수정으로 로그인 화면이 즉석 사용자 객체를 만드는 방식에서 벗어나 실제 사용자 저장소와 세션 저장소를 사용하도록 바뀌었다.
4. 이후 외부 DB를 붙일 때도 엔티티 구조와 권한 구조를 유지한 채 저장소 구현만 교체할 수 있다.

### P2로 넘어갈 수 있는 이유

P1에서 필요한 최소 공통 기반이 정리됐기 때문이다.  
이제 커뮤니티와 관리자 기능을 같은 저장 구조 위에서 확장할 수 있다.

## P2 진행 상태

현재 기준으로 P2는 `진행 중` 상태다.

### 현재까지 진행된 P2

1. 커뮤니티 실서비스화
   - 게시글 저장소를 `communityPosts` 기준으로 통합함
   - 게시글 작성/수정/삭제, 댓글 등록/삭제, 조회수 저장을 같은 helper 계층으로 정리함
2. 관리자 운영 기능 고도화
   - 관리자 게시글 탭이 실제 커뮤니티 저장소를 읽도록 바뀜
   - 검색, 분류 필터, CSV 다운로드, 삭제 기능이 실제 데이터 기준으로 동작함

### 왜 이 방향으로 진행했는가

1. P2에서 가장 구현량이 많고 사용자 체감이 큰 영역이 커뮤니티였다.
2. 사용자 화면과 관리자 화면이 서로 다른 게시글 소스를 보면 운영 기능이 계속 목업으로 남게 된다.
3. 그래서 먼저 게시글과 댓글을 단일 저장소로 묶고, 같은 데이터를 양쪽 화면이 함께 바라보게 만들었다.

## AI 작업 기본 원칙

1. 모든 AI는 작업 시작 전에 `ai-development-rules.md`를 먼저 읽는다.
2. 모든 AI는 `project-priority-table.md`를 확인하고 우선순위 안에서 작업한다.
3. 모든 수정사항은 가능하면 푸시와 배포까지 진행하여 실제 검토 가능 상태로 만든다.
4. 설명이 필요하면 비개발자도 이해할 수 있는 표현을 사용한다.
5. 계획이 필요한 작업은 문서로 만들고, 완료 후 삭제한다.

## 버전 관리 원칙

버전 형식은 `AA.BBB.CCC`를 사용한다.

1. `AA`
   - 대버전
   - 사용자가 직접 판단해서 올린다
2. `BBB`
   - 기능 추가 또는 삭제 시 올린다
3. `CCC`
   - 작은 수정, 버그 수정, 문구 수정, 마이너 개선 시 올린다

## 관리자 페이지 버전 기록 기준

관리자 페이지의 버전 기록은 아래 구조를 따른다.

1. 버전 번호
2. 날짜
3. 200자 내외 핵심 수정사항
4. 세부 업데이트 내역

세부 업데이트 내역은 기능별로 나누어 한눈에 볼 수 있어야 한다.

예시:

- 버전: `01.002.003`
- 핵심 수정사항: 메인 홈 강연 일정 추가, 로그아웃 기능 구현, 마이페이지 기본 구조 추가
- 세부 업데이트:
  - 홈: 왕사남 강연 일정 노출
  - 인증: 로그아웃 및 로그인 상태 유지
  - 계정: 마이페이지 생성

## 문서 동기화 규칙

아래 항목이 바뀌면 `kms.md`도 함께 갱신한다.

1. 개발 규칙
2. 우선순위
3. 버전 체계
4. 구현 현황
5. 디자인 테마
6. 관리자 페이지 운영 방식

## KMS 기록 방식

KMS를 수정할 때는 가능하면 아래 구조를 따른다.

1. 무엇이 바뀌었는가
2. 왜 바꾸게 되었는가
3. 어떤 배경과 맥락이 있었는가
4. 이후 어떤 작업과 연결되는가

짧은 주석이나 변경 기록 형태여도 괜찮지만, 나중에 맥락 없이 봐도 이해 가능해야 한다.

## 기능정의서

이 섹션은 KMS의 제1 기능이다.  
다른 개발자가 코드를 직접 열지 않아도, 현재 사이트에 어떤 기능이 어디에 있고 어느 정도 구현되었는지 먼저 파악할 수 있어야 한다.

### 1. 홈

- 위치: 메인 랜딩 페이지
- 실제 라우트: `home`
- 주요 기능:
  - 브랜드 히어로 섹션
  - 공지사항 노출
  - 왕사남 강연 일정 노출
  - 투어 프로그램 노출
  - 칼럼 추천 노출
  - 파트너십 소개
  - 하단 버전 표시
- 구현 상태:
  - 구현됨: 공지, 강연 일정, 투어, 칼럼, 푸터 버전 표시
  - 부분 구현: 칼럼 상세 이동은 공개 칼럼 페이지 기준
- 사용자 목적:
  - 방문자가 서비스 성격과 최신 일정, 주요 콘텐츠를 한눈에 파악한다.
- 비고:
  - 홈 칼럼 섹션은 관리자 발행 칼럼도 함께 반영하도록 연결됨
  - `왕사남 소개` 별도 섹션은 제거됨

### 2. 인증

- 위치: 로그인/회원가입 페이지
- 실제 라우트: `login`, `signup`
- 주요 기능:
  - 로그인
  - 회원가입
  - 관리자 임시 계정 로그인
  - 로그인 상태 유지
  - 로그아웃
- 구현 상태:
  - 부분 구현
- 사용자 목적:
  - 회원과 관리자가 같은 입구에서 계정 상태를 만들고 유지한다.
- 현재 동작 방식:
  - 관리자 계정은 `admin@admin.admin / admin`
  - 일반 회원도 회원가입 시 사용자 저장소에 등록됨
  - 로그인 상태는 `session` 저장소로 유지됨
- 비고:
  - 현재는 GitHub Pages 기준 local-first 구조이며, 비밀번호는 브라우저 안에서 해시 저장됨
  - 이후 외부 DB 연동 시 저장소 구현만 교체하는 방향으로 확장 예정

### 3. 마이페이지

- 위치: 로그인 후 네비게이션의 `마이페이지`
- 실제 라우트: `mypage`
- 주요 기능:
  - 계정 정보 보기
  - 등급 정보 보기
  - 예정 강연 보기
  - 예정 투어 보기
  - 주문 상태 보기
  - 최근 활동 보기
- 구현 상태:
  - 기본 구현
- 사용자 목적:
  - 로그인한 사용자가 자신의 계정 상태와 최근 활동을 한 곳에서 확인한다.
- 비고:
  - 실제 주문 내역/예약 내역 DB 연동은 아직 없음

### 4. 커뮤니티

- 위치: `커뮤니티`
- 실제 라우트: `community`
- 주요 기능:
  - 게시글 목록
  - 게시글 작성
  - 게시글 수정
  - 게시글 삭제
  - 댓글 작성
  - 댓글 삭제
  - 카테고리별 접근 제한
  - 게시글/댓글 저장
  - 조회수 저장
- 구현 상태:
  - 실동작 기반으로 고도화 진행 중
- 사용자 목적:
  - 회원이 질문과 후기, 정보를 남기고 운영자가 같은 흐름 안에서 관리한다.
- 현재 동작 방식:
  - `communityPosts`, `comments` local-first 저장소 기반
- 비고:
  - 게시글 작성/수정/삭제와 댓글 등록/삭제는 동작함
  - 이후 외부 DB 연동 시 helper 계층은 유지하고 저장소 구현만 교체 예정

### 5. 투어 / 강연

- 위치: `투어 프로그램`
- 실제 라우트: `tour`
- 주요 기능:
  - 투어 상세 소개
  - 답사 프로그램 목록
  - 예약 버튼 / 대기자 등록 버튼
- 구현 상태:
  - 부분 구현
- 사용자 목적:
  - 방문자가 강연/답사 프로그램을 이해하고 예약 의사를 갖게 만든다.
- 비고:
  - 별도 `왕사남 소개` 페이지 진입점은 현재 제거됨
  - 예약 접수와 대기자 처리 로직은 아직 없음

### 6. 칼럼

- 위치: `뱅기노자 칼럼`
- 실제 라우트: `column`
- 주요 기능:
  - 공개 칼럼 목록
  - 칼럼 상세 보기
  - 관리자 발행 칼럼 공개 반영
- 구현 상태:
  - 부분 구현
- 사용자 목적:
  - 방문자와 회원이 뱅기노자 칼럼을 읽고 브랜드 신뢰를 쌓는다.
- 현재 동작 방식:
  - 기본 칼럼 데이터 + 관리자 발행 칼럼(`userColumns`) 병합
- 비고:
  - 관리자 발행 칼럼의 HTML 본문도 공개 상세에서 읽을 수 있음

### 7. 책 / 주문

- 위치: `왕의길`, `checkout`
- 실제 라우트: `book`, `checkout`
- 주요 기능:
  - 책 정보 노출
  - 판본 선택
  - 수량 선택
  - 장바구니
  - 체크아웃 UI
- 구현 상태:
  - 부분 구현
- 사용자 목적:
  - 방문자가 책 정보를 확인하고 구매 흐름으로 진입한다.
- 비고:
  - 실제 주문 저장, 결제 연동, 주문 내역 관리는 아직 없음

### 8. 관리자

- 위치: 관리자 페이지
- 실제 라우트: `admin`
- 주요 기능:
  - 대시보드
  - 게시글/칼럼/투어/회원 관리
  - 카테고리 관리
  - 회원 등급 관리
  - 칼럼 작성
  - 개인정보/GDPR/PIPA 관리
  - 버전 기록
  - KMS
- 구현 상태:
  - 부분 구현
- 사용자 목적:
  - 운영자가 콘텐츠, 회원, 정책, 문서 상태를 한 화면에서 관리한다.
- 현재 동작 방식:
  - 운영 문서와 화면을 함께 관리
  - 관리자 발행 칼럼은 공개 사이트에 반영됨
  - 게시글 탭은 실제 커뮤니티 저장소를 읽음
  - KMS 안에서 `기능정의서`, `디자인`, `운영 원칙`을 나눠 볼 수 있음
- 비고:
  - 주문 관리, 일부 운영 액션, 실제 백오피스 기능은 추가 개발 필요

### 9. 문서 / 운영 체계

- 주요 문서:
  - `project-priority-table.md`
  - `ai-development-rules.md`
  - `kms.md`
- 구현 상태:
  - 기본 구축 완료
- 사용자 목적:
  - 여러 AI와 사람이 같은 규칙과 맥락으로 이어서 작업할 수 있게 한다.
- 비고:
  - 새 규칙이 생기면 AI 가이드와 KMS를 같이 갱신해야 함

## 디자인 원칙서

이 섹션은 화면 작업의 기준 문서다.  
새 UI를 만들거나 기존 화면을 바꿀 때, 아래 원칙을 먼저 확인하고 그 위에서 작업한다.

관리자 페이지에서는 `KMS > 디자인` 탭에서 같은 기준을 확인할 수 있어야 한다.

### 1. 브랜드 무드

- 조선 왕실, 궁궐, 기록물, 전시 도록의 분위기를 기본으로 한다.
- 값싼 동양풍 장식보다 절제된 권위와 정적 긴장을 우선한다.
- 화면은 화려함보다 깊이와 밀도를 느끼게 해야 한다.

### 2. 컬러 원칙

- 기본 배경은 짙은 먹색 계열을 유지한다.
- 금색은 강조용 포인트로만 사용하고 남용하지 않는다.
- 주요 본문은 높은 가독성을 유지하는 밝은 중성 톤을 쓴다.
- 위험/삭제/경고 색상은 금색과 명확히 구분되게 유지한다.

### 3. 타이포그래피 원칙

- 제목은 세리프 중심으로 품격 있게 보이도록 유지한다.
- 본문은 읽기 쉬운 한글 폰트를 사용한다.
- 정보 라벨, 메타 정보, 운영 수치는 모노 계열을 사용해 구조를 분명하게 한다.
- 한 화면 안에서도 제목, 본문, 메타 정보의 역할 차이가 시각적으로 분명해야 한다.

### 4. 레이아웃 원칙

- 여백은 넉넉하게 두고, 답답하게 붙이지 않는다.
- 카드, 표, 섹션은 편집 디자인처럼 정렬감 있게 구성한다.
- 모바일에서도 구조가 무너지지 않도록 한 줄 정보량을 조절한다.
- 기능이 많아도 화면은 조용하고 차분하게 읽혀야 한다.

### 5. 컴포넌트 원칙

- 버튼은 역할이 분명해야 한다.
- 주요 행동은 금색 강조 버튼, 일반 행동은 기본 버튼, 위험 행동은 경고 색상 버튼으로 구분한다.
- 표와 카드의 역할이 겹치지 않게 사용한다.
- 배지, 필터, 상태 라벨은 운영 판단에 도움이 되도록 짧고 명확하게 만든다.

### 6. 인터랙션 원칙

- 애니메이션은 과하지 않게 사용한다.
- hover, focus, active 상태는 분명하되 시끄럽지 않게 표현한다.
- 관리자 화면은 장식보다 정보 탐색성과 작업 효율을 우선한다.
- 사용자 화면은 감성 전달과 신뢰감을 함께 주어야 한다.

### 7. 디자인 금지 원칙

- 밝은 흰색 바탕 중심의 일반 SaaS 느낌으로 바꾸지 않는다.
- 보라색 계열을 브랜드 주색처럼 사용하지 않는다.
- 과한 그라데이션과 유행형 마이크로 인터랙션을 남발하지 않는다.
- 기존 브랜드 분위기와 맞지 않는 귀여운 아이콘 중심 화면으로 흐르지 않게 한다.

## 구현 현황 메모

현재 기준으로 아래 항목은 운영 문서와 연결해서 관리해야 한다.

1. 관리자 KMS 페이지 기본 UI 반영 완료
   - 계기: 저장소 문서와 관리자 화면의 기준이 서로 어긋나면 운영 중 혼선이 생길 수 있었음
   - 배경: 여러 AI가 함께 개발하는 구조에서, 관리자 화면에서도 같은 규칙을 바로 확인할 수 있어야 했음
2. 관리자 버전 기록 UI 기본 반영 완료
   - 계기: 수정사항을 한눈에 파악할 수 있는 운영 화면이 필요했음
   - 배경: 핵심 요약과 세부 업데이트 내역, 그리고 수정 배경까지 함께 보는 구조가 요구되었음
3. 버전 체계를 `AA.BBB.CCC` 형식으로 코드에 반영 완료
   - 계기: 버전 기준이 프로젝트 전반에서 일관되지 않았음
   - 배경: 기능 추가/삭제와 마이너 수정의 차이를 명확히 나눠 관리할 필요가 있었음
4. 배포 완료 상태 확인 절차는 계속 필요
   - 계기: 수정 후 실제 검토 가능 상태를 보장하려면 배포 확인까지 필요함
   - 배경: 현재 GitHub 인증 문제로 자동 푸시가 막힐 수 있으므로, 배포 확인 절차를 계속 운영 규칙으로 유지해야 함
5. 관리자 발행 칼럼과 공개 페이지 연결 기본 반영 완료
   - 계기: 우선순위 P1에서 가장 먼저 해결해야 할 공개 연결 문제가 남아 있었음
   - 배경: 관리자에서 발행한 칼럼이 공개 칼럼 목록과 홈 화면에 보이지 않아, 운영 기능이 실제 사이트에 반영되지 않는 상태였음
6. 하단 버전 표시 강화 완료
   - 계기: 사용자가 커밋, 푸시, 배포 반영 여부를 화면에서 바로 확인하고 싶어 했음
   - 배경: 기존 푸터 버전 표시는 너무 작아서 배포 검토 기준으로 쓰기 어려웠음
7. KMS 기능 인벤토리 확장 반영 완료
   - 계기: 다른 개발자가 코드를 직접 보지 않아도 전체 기능 구조를 이해할 수 있어야 했음
   - 배경: KMS가 규칙 문서 역할만 하고 있어서, 실제 사이트 기능 목록과 구현 상태를 운영 문서 차원에서 읽기 어렵다는 문제가 있었음
8. P1 진행 상태 명시 반영 완료
   - 계기: 일부 P1 작업이 진행되었지만 전체 완료와 부분 완료를 구분해서 기록할 필요가 있었음
   - 배경: 우선순위 규칙상 P2로 넘어가기 전에, 현재 어떤 P1이 끝났고 어떤 P1이 남았는지를 문서에 명확히 남겨야 했음
9. P1 local-first 인증/저장 구조 완료 반영
   - 계기: 로그인과 회원가입이 실제 사용자 저장소를 거치지 않으면 P1을 완료로 보기 어려웠음
   - 배경: GitHub Pages 정적 배포 구조에서도 확장 가능한 저장 계층을 먼저 고정해야 이후 P2 기능이 흔들리지 않음
10. P2 커뮤니티 실서비스화 1차 반영
   - 계기: 커뮤니티는 UI는 많았지만 실제 운영 가능한 수정/삭제/조회수/관리자 연결이 부족했음
   - 배경: 사용자 화면과 관리자 화면이 같은 게시글 저장소를 보지 않으면 운영 기능이 계속 목업 상태에 머물 가능성이 높았음
11. KMS 기능정의서 우선 원칙 및 디자인 원칙서 반영
   - 계기: KMS를 먼저 봤을 때 규칙보다 기능 범위와 디자인 기준이 바로 보여야 다음 개발자가 작업하기 쉬움
   - 배경: 운영 문서가 많아질수록 “무슨 기능이 있는지”와 “어떤 디자인으로 만들어야 하는지”가 첫 화면에서 보이지 않으면 컨텍스트 복원이 느려질 수 있음
12. 대시보드 실데이터 연결 및 KMS 내부 탭 구조 재정리
   - 계기: 관리자 대시보드가 하드코딩 수치로 보이면 실제 운영 판단 기준으로 쓰기 어려움
   - 배경: KMS도 상위 탭 하나로만 보이면 기능정의서와 디자인 기준을 바로 찾기 어려워, KMS 내부에서 목적별로 분리할 필요가 생김
