# KMS

## 목적

이 문서는 `왕사들(WANGSADEUL)` 프로젝트의 운영 기준 문서다.  
현재 구현 현황, 개발 규칙, 버전 관리 원칙, 디자인 방향, AI 협업 기준을 한 곳에서 확인할 수 있어야 한다.

## 문서 운영 원칙

1. `kms.md`와 `ai-development-rules.md`는 같이 관리한다.
2. 새 규칙이 추가되면 두 문서를 동시에 갱신한다.
3. 구현 현황이 바뀌면 KMS도 함께 갱신한다.
4. 관리자 페이지의 KMS 화면은 이 문서와 같은 기준을 반영해야 한다.
5. KMS를 수정할 때는 무엇이 바뀌었는지만 적지 말고, 왜 수정했는지 계기와 배경도 함께 기록한다.
6. KMS 기록은 나중에 다른 AI나 사람이 보고도 맥락을 이해할 수 있어야 한다.

## 현재 우선순위 기준

기준 문서: `project-priority-table.md`

1. 인증/권한
2. 데이터 저장 구조
3. 관리자 발행물과 공개 페이지 연결
4. 커뮤니티 실서비스화
5. 관리자 운영 기능 고도화
6. 투어 예약
7. 책 주문/결제
8. 부가 기능

## AI 작업 기본 원칙

1. 모든 AI는 작업 시작 전에 `ai-development-rules.md`를 먼저 읽는다.
2. 모든 AI는 `project-priority-table.md`를 확인하고 우선순위 안에서 작업한다.
3. 모든 수정사항은 가능하면 푸시와 배포까지 진행하여 실제 검토 가능 상태로 만든다.
4. 설명이 필요하면 비개발자도 이해할 수 있는 표현을 사용한다.
5. 계획이 필요한 작업은 문서로 만들고, 완료 후 삭제한다.

## 버전 관리 원칙

버전 형식은 `AA.BBB.CCC`를 사용한다.

1. `AA`
   - 대버전
   - 사용자가 직접 판단해서 올린다
2. `BBB`
   - 기능 추가 또는 삭제 시 올린다
3. `CCC`
   - 작은 수정, 버그 수정, 문구 수정, 마이너 개선 시 올린다

## 관리자 페이지 버전 기록 기준

관리자 페이지의 버전 기록은 아래 구조를 따른다.

1. 버전 번호
2. 날짜
3. 200자 내외 핵심 수정사항
4. 세부 업데이트 내역

세부 업데이트 내역은 기능별로 나누어 한눈에 볼 수 있어야 한다.

예시:

- 버전: `01.002.003`
- 핵심 수정사항: 메인 홈 강연 일정 추가, 로그아웃 기능 구현, 마이페이지 기본 구조 추가
- 세부 업데이트:
  - 홈: 왕사남 강연 일정 노출
  - 인증: 로그아웃 및 로그인 상태 유지
  - 계정: 마이페이지 생성

## KMS 기록 방식

KMS를 수정할 때는 가능하면 아래 구조를 따른다.

1. 무엇이 바뀌었는가
2. 왜 바꾸게 되었는가
3. 어떤 배경과 맥락이 있었는가
4. 이후 어떤 작업과 연결되는가

짧은 주석이나 변경 기록 형태여도 괜찮지만, 나중에 맥락 없이 봐도 이해 가능해야 한다.

## 사이트 기능 인벤토리

이 섹션은 다른 개발자가 코드를 직접 열지 않아도, 현재 사이트에 어떤 기능이 어디에 있고 어느 정도 구현되었는지 파악할 수 있도록 만든 기능 목록이다.

### 1. 홈

- 위치: 메인 랜딩 페이지
- 주요 기능:
  - 브랜드 히어로 섹션
  - 공지사항 노출
  - 왕사남 강연 일정 노출
  - 투어 프로그램 노출
  - 칼럼 추천 노출
  - 파트너십 소개
  - 하단 버전 표시
- 구현 상태:
  - 구현됨: 공지, 강연 일정, 투어, 칼럼, 푸터 버전 표시
  - 부분 구현: 칼럼 상세 이동은 공개 칼럼 페이지 기준
- 비고:
  - 홈 칼럼 섹션은 관리자 발행 칼럼도 함께 반영하도록 연결됨

### 2. 인증

- 위치: 로그인/회원가입 페이지
- 주요 기능:
  - 로그인
  - 회원가입
  - 관리자 임시 계정 로그인
  - 로그인 상태 유지
  - 로그아웃
- 구현 상태:
  - 부분 구현
- 현재 동작 방식:
  - 관리자 계정은 `admin@admin.admin / admin`
  - 일반 회원은 임시 상태 저장 방식
  - 로그인 상태는 `localStorage`에 저장됨
- 비고:
  - 실제 서버 인증, 비밀번호 해시, 계정 검증은 아직 없음

### 3. 마이페이지

- 위치: 로그인 후 네비게이션의 `마이페이지`
- 주요 기능:
  - 계정 정보 보기
  - 등급 정보 보기
  - 예정 강연 보기
  - 예정 투어 보기
  - 주문 상태 보기
  - 최근 활동 보기
- 구현 상태:
  - 기본 구현
- 비고:
  - 실제 주문 내역/예약 내역 DB 연동은 아직 없음

### 4. 커뮤니티

- 위치: `커뮤니티`
- 주요 기능:
  - 게시글 목록
  - 게시글 작성
  - 댓글 작성
  - 카테고리별 접근 제한
  - 게시글/댓글 저장
- 구현 상태:
  - 부분 구현
- 현재 동작 방식:
  - `localStorage` 기반
- 비고:
  - 서버 저장, 수정/삭제 정책 고도화는 후속 작업 필요

### 5. 왕사남 소개 / 투어

- 위치: `왕사남 소개`, `투어 프로그램`
- 주요 기능:
  - 왕사남 팀 소개
  - 투어 상세 소개
  - 답사 프로그램 목록
  - 예약 버튼 / 대기자 등록 버튼
- 구현 상태:
  - 부분 구현
- 비고:
  - 예약 접수와 대기자 처리 로직은 아직 없음

### 6. 칼럼

- 위치: `뱅기노자 칼럼`
- 주요 기능:
  - 공개 칼럼 목록
  - 칼럼 상세 보기
  - 관리자 발행 칼럼 공개 반영
- 구현 상태:
  - 부분 구현
- 현재 동작 방식:
  - 기본 칼럼 데이터 + 관리자 발행 칼럼(`userColumns`) 병합
- 비고:
  - 관리자 발행 칼럼의 HTML 본문도 공개 상세에서 읽을 수 있음

### 7. 책 / 주문

- 위치: `왕의길`, `checkout`
- 주요 기능:
  - 책 정보 노출
  - 판본 선택
  - 수량 선택
  - 장바구니
  - 체크아웃 UI
- 구현 상태:
  - 부분 구현
- 비고:
  - 실제 주문 저장, 결제 연동, 주문 내역 관리는 아직 없음

### 8. 관리자

- 위치: 관리자 페이지
- 주요 기능:
  - 대시보드
  - 게시글/칼럼/투어/회원 관리
  - 카테고리 관리
  - 회원 등급 관리
  - 칼럼 작성
  - 개인정보/GDPR/PIPA 관리
  - 버전 기록
  - KMS
- 구현 상태:
  - 부분 구현
- 현재 동작 방식:
  - 운영 문서와 화면을 함께 관리
  - 관리자 발행 칼럼은 공개 사이트에 반영됨
- 비고:
  - 주문 관리, 일부 운영 액션, 실제 백오피스 기능은 추가 개발 필요

### 9. 문서 / 운영 체계

- 주요 문서:
  - `project-priority-table.md`
  - `ai-development-rules.md`
  - `kms.md`
- 구현 상태:
  - 기본 구축 완료
- 비고:
  - 새 규칙이 생기면 AI 가이드와 KMS를 같이 갱신해야 함

## 구현 현황 메모

현재 기준으로 아래 항목은 운영 문서와 연결해서 관리해야 한다.

1. 관리자 KMS 페이지 기본 UI 반영 완료
   - 계기: 저장소 문서와 관리자 화면의 기준이 서로 어긋나면 운영 중 혼선이 생길 수 있었음
   - 배경: 여러 AI가 함께 개발하는 구조에서, 관리자 화면에서도 같은 규칙을 바로 확인할 수 있어야 했음
2. 관리자 버전 기록 UI 기본 반영 완료
   - 계기: 수정사항을 한눈에 파악할 수 있는 운영 화면이 필요했음
   - 배경: 핵심 요약과 세부 업데이트 내역, 그리고 수정 배경까지 함께 보는 구조가 요구되었음
3. 버전 체계를 `AA.BBB.CCC` 형식으로 코드에 반영 완료
   - 계기: 버전 기준이 프로젝트 전반에서 일관되지 않았음
   - 배경: 기능 추가/삭제와 마이너 수정의 차이를 명확히 나눠 관리할 필요가 있었음
4. 배포 완료 상태 확인 절차는 계속 필요
   - 계기: 수정 후 실제 검토 가능 상태를 보장하려면 배포 확인까지 필요함
   - 배경: 현재 GitHub 인증 문제로 자동 푸시가 막힐 수 있으므로, 배포 확인 절차를 계속 운영 규칙으로 유지해야 함
5. 관리자 발행 칼럼과 공개 페이지 연결 기본 반영 완료
   - 계기: 우선순위 P1에서 가장 먼저 해결해야 할 공개 연결 문제가 남아 있었음
   - 배경: 관리자에서 발행한 칼럼이 공개 칼럼 목록과 홈 화면에 보이지 않아, 운영 기능이 실제 사이트에 반영되지 않는 상태였음
6. 하단 버전 표시 강화 완료
   - 계기: 사용자가 커밋, 푸시, 배포 반영 여부를 화면에서 바로 확인하고 싶어 했음
   - 배경: 기존 푸터 버전 표시는 너무 작아서 배포 검토 기준으로 쓰기 어려웠음
7. KMS 기능 인벤토리 확장 반영 완료
   - 계기: 다른 개발자가 코드를 직접 보지 않아도 전체 기능 구조를 이해할 수 있어야 했음
   - 배경: KMS가 규칙 문서 역할만 하고 있어서, 실제 사이트 기능 목록과 구현 상태를 운영 문서 차원에서 읽기 어렵다는 문제가 있었음
